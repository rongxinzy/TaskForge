TaskForge 产品需求文档 PRD

版本：v0.1
阶段：立项版 / MVP 定义版
产品定位：本地 Runner 优先的 AI 原生工程协作工作台
核心原则：数据库驱动业务控制面，Local Runner 执行代码任务，不做云端执行，不以 Kubernetes Operator 作为业务对象实现方式。

⸻

1. 产品定义

TaskForge 是一个面向工程团队的 AI 原生工程协作工作台。平台围绕项目、需求、Bug、技术债、自动扫描问题和 Sprint 工作项组织研发流程，并允许工程师在每个工作项中选择本地 Agent Runner 启动执行会话。平台负责任务上下文组织、Runner 调度、会话状态管理、实时过程可视化、产物归档、审计追踪和团队协作；代码读取、文件修改、命令执行、测试运行和 Agent CLI 调用全部发生在工程师本地机器上的 Local Runner 中。

TaskForge 第一版不构建云端执行环境，不托管用户源码执行，不创建远程沙箱，不实现 Kubernetes Runner，不做 Runner Pool 调度。产品的核心价值不是替代 Codex CLI、Claude Code、OpenClaw 等 Agent 工具，而是把原本分散在 Issue、PR、聊天记录、本地终端、Agent 会话、CI 日志和个人经验中的工程任务执行过程统一纳入团队可见、可审计、可复用的协作工作流。

一句话定义：

TaskForge 让每个工程任务都能带着完整上下文被本地 Agent Runner 接手，并让团队实时看见执行过程、产物和结论。

⸻

2. 产品背景与问题

现有工程团队通常已经使用 GitHub、GitLab、Jira、飞书、Slack、Notion、Confluence、CI/CD、PR Review 和各类本地 Agent CLI 工具完成研发协作，但这些系统之间缺少以“任务执行”为中心的统一控制面。需求、Bug 和技术债通常记录在 Issue 或项目管理工具中，任务上下文散落在评论、文档、提交记录、排查记录、PR、CI 日志和个人记忆中。工程师真正开始处理任务时，需要手动查找上下文、整理复现路径、定位相关代码、选择本地工具、编写 Prompt、运行 Agent，再把结果通过 PR、评论或聊天工具回传给团队。

Agent 工具解决了“执行代码修改”的一部分问题，但它们通常是个人本地工具。团队成员难以知道某个任务是否已经交给 Agent 处理、Agent 当前处于什么状态、使用了哪些上下文、执行了哪些命令、改了哪些文件、测试是否通过、失败原因是什么，以及本次执行中的 Prompt、排查路径和经验是否能够复用。结果是 Agent 执行能力存在于个人工作站中，而团队协作系统仍停留在人工状态流转和结果性同步上。

TaskForge 要解决的问题不是“再做一个 Coding Agent”，而是建立一个团队级任务执行控制面，使每个 WorkItem 在进入执行前拥有可审查的 ContextBundle，在执行中产生可观察的 AgentSession，在执行后沉淀可追溯的 Artifact、Prompt、日志、Diff、验证结果和经验记录。第一版以 Local Runner 作为唯一执行形态，避免云端执行带来的安全、成本、隔离、凭据和环境复杂度，优先验证工程协作、上下文编译、会话可观测和本地 Agent 接入的核心闭环。

⸻

3. 产品目标

TaskForge v0.1 的目标是打通从项目工作空间到本地 Agent 执行会话的完整闭环。用户进入平台后，可以选择项目卡片进入项目空间，查看该项目已经关联的 GitHub 或 GitLab 仓库，并在项目内查看 Roadmap、需求清单、自动扫描问题卡片和协作看板。用户在协作看板中选择一个属于自己的需求、Bug 或技术债工作项，进入工作项详情页后，可以看到该任务的完整上下文，包括来源、需求关联、问题描述、验收标准、排查记录、相关文件、相关 Issue、PR、提交记录、历史执行会话和推荐验证命令。随后用户可以选择一个在线的本地 Runner，例如本机 Codex CLI Runner，并以默认 /goal 模式启动任务执行。启动后，Local Runner 在用户本地工作区调用 Agent CLI 执行任务，并实时向平台回传会话事件。其他项目成员可以在看板和会话页面看到该任务进入执行状态，并可进入 Session 页面查看执行过程、状态、日志、Diff、测试结果和最终产物。

本版本必须达成三个产品结果。第一，工程任务不再只是静态卡片，而是可以被编译成带上下文的可执行对象。第二，本地 Agent 执行不再是个人黑盒，而是成为团队可观察的 Session。第三，每次执行都能沉淀 Prompt、上下文、事件流、产物和结果，为后续复用、审计和团队协作提供依据。

⸻

4. 非目标

TaskForge v0.1 明确不做云端执行。平台不在云端 clone 私有仓库，不在云端运行用户代码，不创建云端沙箱，不提供远程 Agent 执行池，不进行 Runner 自动调度，也不管理 GPU、Kubernetes Job 或容器化执行环境。所有代码读写、命令执行、测试运行和 Agent CLI 调用必须发生在用户本地机器上的 Local Runner 中。

TaskForge v0.1 不做完整项目管理套件。Roadmap、需求清单、看板和协作功能服务于 AI 任务执行闭环，不以替代 Jira、Linear、GitHub Projects 或 GitLab Issues 为目标。第一版允许创建、导入、展示和关联 Roadmap、Requirement、WorkItem，但不做复杂资源排期、燃尽图、容量规划、跨团队 Portfolio 管理和复杂报表。

TaskForge v0.1 不做通用知识库，不做 IM 机器人，不做邮件自动化，不做多 Agent 炫技编排，不做 Skill Marketplace，不做企业级多租户执行池，不做 Kubernetes Operator 业务控制面，不做云端源码索引服务。MCP、Skill、云端 Runner、K8s Runner、企业 SSO、RBAC、Runner Pool 和 InferenceLab 扩展可以作为后续阶段能力，但不得进入第一版主路径。

6. 核心使用场景

用户打开 TaskForge 平台后，首页展示其有权限访问的项目卡片。用户选择一个项目进入项目工作空间，平台展示该项目已经链接的 GitHub 或 GitLab 仓库，以及项目的 Roadmap、需求清单、自动扫描问题卡片、协作看板、执行会话和设置入口。用户进入协作看板，看到当前 Sprint 中的需求、Bug 和技术债，每张卡片展示任务类型、标题、负责人、优先级、当前状态、上下文完整度、是否存在活跃 Session、Runner 状态、PR 状态和验证状态。

用户选择一个属于自己的 Bug 卡片进入详情页。详情页展示任务标题、来源、所属 Sprint、负责人、当前状态、验收标准、需求关联、问题描述、复现步骤、排查记录、相关文件、相关 PR、相关提交、历史 Session 和平台生成的 ContextBundle。页面右侧展示执行控制区，用户可以选择一个在线 Local Runner，例如 saber-local-codex，执行模式默认为 /goal。用户可以直接点击 Start，也可以在底部命令输入框中输入 /goal 指令补充约束，例如要求优先不修改公共 API 或先补充回归测试。

用户启动任务后，平台创建 AgentSession，编译 ContextBundle，生成 ExecutionEnvelope，并通过 Runner Gateway 将执行任务下发给用户本地 Local Runner。Local Runner 接收任务后检查本地仓库绑定、Git 状态、Agent CLI 是否可用、命令权限和敏感路径规则，然后在本地工作区创建临时分支或工作树，并调用 Codex CLI 执行任务。执行过程中，Local Runner 将 Agent 输出、命令执行、文件变更、Diff、测试结果、错误、审批请求和最终总结以 SessionEvent 形式实时回传平台。

其他团队成员在协作看板中看到该任务状态进入进行中，并显示存在一个 Running Session。成员可以点击进入 Session 页面，查看该任务的目标、上下文包、Runner、执行模式、事件时间线、命令日志、文件变更、Diff、验证结果和最终产物。执行完成后，Session 状态变为 Completed 或 Failed。若产生可用 Patch 或 PR，WorkItem 状态进入 Needs Review；若执行失败，WorkItem 保持 In Progress 或 Blocked，并展示失败原因、日志和可重试入口。

7. 产品信息架构

TaskForge v0.1 的一级信息结构由 Project Workspace 承载。每个 Project Workspace 至少包含 Overview、Roadmap、Requirements、Findings、Board、Sessions、Knowledge 和 Settings 八个入口。Overview 展示项目健康状态、已连接仓库、在线 Runner、当前 Sprint、活跃 Session、近期失败任务和待 Review 任务。Roadmap 以简单列表或分组方式展示项目目标和阶段性计划，并允许关联 Requirement。Requirements 展示需求清单，并支持将需求拆分或关联到 WorkItem。Findings 展示平台自动扫描出来的问题卡片，并支持将 Finding 转换为 WorkItem、关联已有 WorkItem、忽略或标记误报。Board 是第一版核心页面，用于展示当前 Sprint 和 Backlog 中的 WorkItem。Sessions 展示项目内所有活跃和历史 AgentSession。Knowledge 展示执行沉淀，包括 Prompt 版本、上下文包、排查记录、Session 总结和可复用经验。Settings 负责项目成员、仓库连接、Runner 配置、权限和安全规则。

TaskForge 的二级工作空间是 WorkItem Detail。该页面是用户从协作流程进入 Agent 执行流程的关键入口。页面必须同时呈现任务上下文、执行控制、团队评论和历史 Session。用户在该页面完成上下文审查、Runner 选择、模式选择、执行启动、人工补充指令、审批响应和执行结果查看。

TaskForge 的三级工作空间是 AgentSession Viewer。该页面用于展示一次 Agent 执行会话的完整过程。Session Viewer 必须支持实时事件流、历史回放、事件过滤、命令输出查看、Diff 查看、Artifact 下载、验证结果查看、失败原因查看和人工输入记录查看。Session Viewer 是团队成员“围观”和审计本地 Agent 执行过程的核心界面。

8. 核心对象模型

TaskForge v0.1 的核心对象为 Project、Repository、RoadmapItem、Requirement、Finding、WorkItem、ContextBundle、RunnerProfile、ExecutionEnvelope、AgentSession、SessionEvent、Artifact、PromptVersion 和 AuditLog。数据库是这些业务对象的唯一事实来源。Local Runner 只执行任务，不拥有业务状态的最终解释权。

对象	定义	关键关系
Project	一个工程项目的协作空间	拥有 Repository、Requirement、WorkItem、Finding、Runner、Session
Repository	与项目关联的 GitHub 或 GitLab 仓库	属于 Project，可被 WorkItem、Finding、Runner 绑定
RoadmapItem	项目阶段性目标或版本计划	可关联多个 Requirement
Requirement	需求条目	可拆分或关联多个 WorkItem
Finding	自动扫描发现的问题卡片	可转换为 WorkItem 或关联已有 WorkItem
WorkItem	协作看板上的任务对象，可代表需求、Bug、技术债或扫描问题	绑定 ContextBundle、AgentSession、Artifact
ContextBundle	为某个 WorkItem 编译得到的执行上下文包	被 AgentSession 引用
RunnerProfile	本地 Runner 在平台中的注册信息	可执行 AgentSession
ExecutionEnvelope	平台下发给 Local Runner 的结构化执行信封	由 ContextBundle、权限、目标和验证命令组成
AgentSession	一次 Runner 对 WorkItem 的执行尝试	产生 SessionEvent 和 Artifact
SessionEvent	AgentSession 的追加式事件记录	用于实时展示、回放和审计
Artifact	Session 产生的产物，如 Patch、日志、测试报告、PR 链接	归属于 AgentSession
PromptVersion	/goal、/plan 等模式使用的 Prompt 模板版本	被 ContextBundle 或 AgentSession 引用
AuditLog	用户、平台和 Runner 的关键操作记录	用于安全审计

WorkItem 是产品协作层的核心对象，AgentSession 是执行层的核心对象，ContextBundle 是两者之间的桥梁。所有需求、Bug、技术债和自动扫描问题最终都应进入 WorkItem 流程。所有本地 Agent 执行都必须产生 AgentSession。所有 AgentSession 都必须引用一个 ContextBundle，并持续追加 SessionEvent。

⸻

9. 状态机定义

WorkItem 表示团队协作状态，不直接等同于 Agent 执行状态。WorkItem 第一版状态包括 Backlog、Ready、In Progress、Blocked、Needs Review、Done 和 Cancelled。一个 WorkItem 在被用户领取或进入 Sprint 后可以进入 In Progress；当用户启动本地 Runner 后，该 WorkItem 仍为 In Progress，但其 Active Session 状态变为 Running；当 Agent 执行完成并产生 Patch 或 PR 后，WorkItem 进入 Needs Review；当 Review 完成并满足验收标准后进入 Done；当执行失败且需要外部信息、环境或人工决策时进入 Blocked。

AgentSession 表示单次执行尝试的状态。第一版状态包括 Created、Context Compiling、Queued、Dispatching、Running、Awaiting Input、Awaiting Approval、Verifying、Completed、Failed、Cancelled 和 Interrupted。Interrupted 专门用于 Local Runner 掉线、用户电脑休眠、网络中断或 Runner 进程异常退出。Completed 只表示本次 Session 正常结束，不必然表示 WorkItem 已完成；Failed 表示本次 Session 未能达成目标；Verifying 表示 Local Runner 正在运行验证命令或收集验证结果。

RunnerProfile 表示 Local Runner 在线状态。第一版状态包括 Offline、Online、Busy、Error 和 Disabled。Runner 启动后主动连接平台并上报 Online；接收 Session 后进入 Busy；执行结束后恢复 Online；本地 Agent CLI 不可用、本地仓库绑定异常、权限校验失败或运行时异常时进入 Error；管理员或 Runner Owner 可以将 Runner 设置为 Disabled。

Finding 表示自动扫描问题状态。第一版状态包括 Open、Converted、Linked、Ignored、False Positive 和 Snoozed。Open 表示问题待处理；Converted 表示已转为 WorkItem；Linked 表示已关联到已有 WorkItem；Ignored 表示团队决定不处理；False Positive 表示误报；Snoozed 表示延后处理。

10. 功能需求

10.1 项目空间

项目空间必须支持项目卡片列表、项目详情页、成员列表、仓库连接状态、当前 Sprint 概览、活跃 Session 概览和基础设置。用户只能看到其有权限访问的项目。项目卡片应展示项目名称、描述、关联仓库数量、当前活跃 WorkItem 数、运行中 Session 数和最近更新时间。

项目详情页必须提供固定标签页：Overview、Roadmap、Requirements、Findings、Board、Sessions、Knowledge 和 Settings。第一版允许 Roadmap 和 Requirements 以简单列表形式存在，不要求支持复杂层级、跨项目依赖和高级排期，但必须支持与 WorkItem 的关联关系。Settings 中必须支持 GitHub/GitLab 仓库连接、成员管理、Runner 可见性配置、敏感路径规则和命令白名单配置。

10.2 GitHub / GitLab 仓库连接

平台必须允许项目连接一个或多个 GitHub/GitLab 仓库。第一版至少支持通过用户提供的访问令牌或 OAuth App 读取仓库元数据、Issue、Pull Request 或 Merge Request、提交记录和基础 CI 状态。平台不得在云端执行仓库代码，也不得在云端运行测试。仓库连接用于组织上下文、关联 WorkItem、导入 Issue、读取 PR/MR 状态和展示代码引用。

仓库对象必须记录 provider、url、default_branch、external_id、连接状态、最近同步时间和同步错误。若仓库连接失效，项目空间必须展示明确错误，并禁止基于该仓库创建新的自动上下文引用，但不得影响已有 WorkItem 和 Session 历史查看。

10.3 Roadmap 与需求清单

Roadmap 第一版用于记录项目阶段性目标。每个 RoadmapItem 至少包含标题、描述、状态、目标日期、负责人和关联 Requirement。Requirement 用于记录需求清单，每个 Requirement 至少包含标题、描述、优先级、状态、验收说明、来源和关联 WorkItem。Roadmap 和 Requirement 不作为第一版执行入口，执行入口必须是 WorkItem。

Requirement 可以手动创建，也可以从 GitHub/GitLab Issue 导入。Requirement 可以拆分为多个 WorkItem，也可以关联已有 WorkItem。Requirement 页面必须展示其下游 WorkItem 的状态分布、活跃 Session、PR/MR 链接和完成情况。

10.4 自动扫描问题卡片 Findings

平台第一版需要提供基础自动扫描能力，用于生成 Findings。v0.1 的 Findings 类型限定为 CI 配置问题、文档漂移和测试缺口三类。CI 配置问题包括缺少依赖缓存、无并发取消、测试和构建阶段混杂、明显重复 Job 等。文档漂移包括 README 中的安装命令、启动命令、CLI 参数、环境变量和实际仓库文件不一致。测试缺口包括高变更模块无测试文件、核心路径缺少测试引用、Bug 类 WorkItem 无回归测试关联等。

Finding 不是直接执行对象。每个 Finding 必须经过人工 Triage 后才能转为 WorkItem 或关联到已有 WorkItem。Finding 卡片必须展示标题、类型、严重级别、证据、建议操作和状态，并提供 Convert to WorkItem、Link to WorkItem、Ignore、Snooze、Mark False Positive 操作。Finding 转为 WorkItem 后，原始证据必须进入 WorkItem 的 ContextBundle 候选上下文。

10.5 协作看板 Board

Board 是第一版核心页面。Board 必须支持按 Sprint 展示 WorkItem，并至少提供 Backlog、Ready、In Progress、Blocked、Needs Review、Done 六列视图。每张 WorkItem 卡片必须展示类型、标题、负责人、优先级、状态、上下文完整度、是否存在 Active Session、Runner 名称、Session 状态、PR/MR 状态和验证状态。卡片必须能够点击进入 WorkItem Detail。

Board 必须支持基础筛选，包括负责人、类型、优先级、状态、是否有 Active Session、是否 Needs Review、是否 Blocked。Board 必须支持拖拽或菜单方式修改 WorkItem 状态，但当 WorkItem 存在 Running Session 时，不允许直接标记 Done；必须等待 Session 完成并进入 Review 或由有权限用户强制终止 Session 后才可关闭任务。

10.6 WorkItem 详情页

WorkItem Detail 是任务执行入口。页面必须展示任务基本信息、来源、所属 Requirement、所属 Sprint、负责人、优先级、状态、描述、验收标准、复现步骤、排查记录、相关文件、相关 Issue、相关 PR/MR、相关提交、历史 Session、评论、ContextBundle 和执行控制区。

执行控制区必须允许用户选择一个在线 Local Runner，并选择执行模式。v0.1 默认执行模式为 /goal，同时支持 /plan 和 /investigate。/goal 表示允许 Agent 在权限边界内读取文件、修改文件、运行允许命令并尝试完成任务目标。/plan 表示只生成执行计划，不允许修改代码。/investigate 表示只进行排查和证据整理，不允许修改代码。第一版可以通过 Runner Adapter 层强制执行模式约束，若某个 Adapter 无法严格限制行为，界面必须明确提示该限制。

WorkItem 底部必须提供输入区，但必须区分团队评论和 Agent 指令。普通评论不进入当前 AgentSession 的即时输入，除非用户明确选择“发送给 Agent”或使用斜杠命令。斜杠命令包括 /goal、/plan、/investigate、/ask、/pause、/resume、/stop、/approve 和 /reject。所有进入 Agent 的人工输入必须作为 SessionEvent 记录，并在 Session 页面可见。

10.7 ContextBundle 上下文包

ContextBundle 是 WorkItem 执行前由平台编译的上下文对象。它必须包含任务摘要、目标、验收标准、来源信息、相关需求、相关 Finding 证据、复现步骤、排查记录、相关文件路径、相关 Issue、相关 PR/MR、相关提交、历史 Session 摘要、推荐验证命令和生成 Prompt。ContextBundle 必须有版本号。每次启动新的 AgentSession 时，平台必须生成或选择一个 ContextBundle 版本，并将其绑定到该 Session。

ContextBundle 第一版不要求保存仓库完整源码。默认只保存文件路径、代码引用、摘要和必要的片段。对于需要传递给 Local Runner 的完整文件内容，优先由 Local Runner 在本地仓库读取。平台不得要求上传整个仓库内容。ContextBundle 必须标记 token 估算、上下文来源和生成时间。当 WorkItem 关键信息变更、仓库同步发生重大变化或历史 Session 产生新结论后，旧 ContextBundle 应标记为 Stale，并允许用户重新编译。

10.8 Local Runner 注册与管理

Local Runner 是 v0.1 唯一执行方式。用户必须在本地安装并运行 Runner Daemon。Runner Daemon 通过登录平台获得凭据，主动与平台建立出站连接，注册 RunnerProfile，上报心跳、能力、Adapter 类型、本地仓库绑定和状态。平台不得主动连接用户本机。

RunnerProfile 至少包含名称、Owner、类型、Adapter、状态、能力、权限、仓库绑定、最近心跳时间和版本号。v0.1 必须支持 Codex CLI Adapter，并建议保留 Shell Adapter 用于诊断。Runner 必须支持绑定项目仓库到本地路径，例如将 GitHub/GitLab 仓库 org/repo 绑定到 /Users/saber/work/repo。启动 Session 前，Runner 必须校验本地路径存在、Git remote 与平台仓库匹配、工作区状态符合策略、Agent CLI 可用、允许命令和敏感路径规则有效。

Runner 必须支持本地事件缓存。当网络中断时，Runner 应将 SessionEvent 写入本地 spool 文件，并在重连后尝试补传。v0.1 不要求支持完整 Session Resume，但必须能够在重连后上报 Interrupted、Failed 或 Completed 状态，并保留本地 Patch 与日志路径供用户处理。

10.9 ExecutionEnvelope 执行信封

平台启动 Session 时必须向 Local Runner 下发结构化 ExecutionEnvelope，而不是仅下发一段 Prompt。ExecutionEnvelope 必须包含 sessionId、workItemId、projectId、执行模式、目标、ContextBundle 引用、仓库信息、本地绑定要求、允许路径、禁止路径、允许命令、验证命令、输出要求和是否允许创建 PR/MR。

ExecutionEnvelope 是平台与 Runner 的核心协议。Local Runner 必须根据 ExecutionEnvelope 生成 Agent CLI 输入，并在本地执行前进行权限校验。若 Runner 无法满足信封要求，例如本地仓库未绑定、命令不在允许列表、Agent CLI 不可用或 Git 工作区不干净，Runner 必须拒绝执行并返回 runner.rejected 事件，平台将 Session 标记为 Failed 或 Awaiting Input。

10.10 AgentSession 与实时会话

AgentSession 表示一次具体执行尝试。一个 WorkItem 可以拥有多个 AgentSession，但同一时间只能有一个 Active Session。Session 启动后，平台必须实时展示 Session 状态、Runner、执行模式、目标、ContextBundle、事件时间线、命令输出、文件变更、Diff、验证结果和 Artifact。项目成员在拥有权限的情况下可以进入 Session 页面查看实时过程。

Session 页面必须支持实时流式更新。第一版可以使用 WebSocket 或 SSE 实现。SessionEvent 必须采用 append-only 模型存储，严禁用单个大 JSON 反复覆盖整个会话过程。事件必须有单调递增 seq，以支持断线重连和历史回放。平台必须支持按事件类型过滤，例如只看命令、只看文件变更、只看验证结果、只看人工输入。

10.11 SessionEvent 事件类型

v0.1 必须支持以下事件类型：session.created、context.compiled、runner.dispatched、runner.accepted、runner.rejected、session.started、agent.input.created、agent.output.delta、agent.output.completed、command.started、command.output、command.finished、file.changed、diff.generated、verification.started、verification.passed、verification.failed、artifact.uploaded、session.awaiting_input、approval.requested、human.input、session.completed、session.failed、session.cancelled、session.interrupted。

事件 payload 必须使用 JSON 格式，并根据事件类型定义最小字段。command.output 必须支持日志脱敏。file.changed 不应默认上传完整文件内容，应至少包含路径、变更类型和摘要。diff.generated 可以作为 Artifact 保存，并在事件中引用 artifactId。所有人工输入、审批、拒绝、停止和恢复操作必须记录操作者、时间和原因。

10.12 Artifact 产物管理

Artifact 表示 Session 产生的可追溯产物。v0.1 必须支持 Patch、Diff、命令日志、测试报告、Agent 总结、PR/MR 链接和错误报告。Artifact 可以存储在本地文件系统或 S3 兼容对象存储中，数据库保存 metadata 和 storage_url。平台必须允许用户从 Session 页面查看或下载 Artifact。

默认策略下，平台保存 Patch、日志、测试报告和总结，不保存完整源码快照。Artifact 上传前必须经过敏感信息过滤。若日志中发现疑似 token、密钥、私钥、.env 内容或被禁止路径内容，Runner 必须进行脱敏或拒绝上传，并在事件中记录 redaction.applied。

10.13 PR/MR 创建

v0.1 可以支持从本地 Runner 创建 PR/MR，但默认不自动执行。默认流程是 Local Runner 生成 Patch 和总结，用户在平台查看后选择 Create PR/MR 或 Export Patch。若开启创建 PR/MR，Runner 应在本地创建分支、提交变更、push 到远端并调用 GitHub/GitLab API 创建 PR/MR。该操作必须受 Runner 本地权限控制，默认需要用户确认。

PR/MR 创建成功后，平台必须记录链接、分支名、提交哈希、目标分支、创建者、SessionId 和 WorkItemId，并将 WorkItem 状态转为 Needs Review。若创建失败，平台必须展示错误原因，并保留 Patch Artifact。

10.14 Prompt 与模式

v0.1 必须内置 /goal、/plan 和 /investigate 三种模式的 Prompt 模板。PromptVersion 必须被记录，并绑定到 AgentSession。平台在生成 ExecutionEnvelope 时应将 WorkItem 目标、验收标准、ContextBundle 摘要、权限约束和验证命令注入 Prompt。Prompt 模板必须可查看版本，但 v0.1 不要求实现完整 Prompt Review 流程。后续版本可以加入 Prompt Review、Prompt Diff、Prompt 成功率统计和团队 Prompt Registry。

/goal 模式要求 Agent 尝试完成任务并输出总结、变更说明、验证结果和后续建议。/plan 模式要求 Agent 输出计划、风险、涉及文件和验证策略，不得修改代码。/investigate 模式要求 Agent 输出排查过程、证据、疑似根因和下一步建议，不得修改代码。

⸻

11. 安全与权限要求

TaskForge v0.1 的安全模型必须明确区分平台权限和 Runner 本地权限。平台权限控制用户是否可以访问项目、查看 WorkItem、启动 Session、查看 Session、评论、配置仓库、配置 Runner、转换 Finding 和修改状态。Runner 本地权限控制 Agent 可以访问哪些本地仓库、修改哪些路径、执行哪些命令、是否允许创建分支、是否允许 push、是否允许创建 PR/MR，以及高风险操作是否需要确认。

平台默认不得保存用户完整源码。Local Runner 默认不得上传 .env、secrets/**、.ssh/**、私钥、token、凭据文件和其他敏感路径内容。Runner 必须提供 deny path 配置，并在执行前对 ExecutionEnvelope 中的 allowedPaths、deniedPaths 和 allowedCommands 进行校验。命令执行默认采用白名单策略；若用户启用宽松模式，界面和日志中必须明确标记该 Runner 为高风险配置。

所有关键操作必须写入 AuditLog，包括项目创建、仓库连接、Runner 注册、Runner 授权、WorkItem 状态变更、Session 启动、Session 停止、人工输入、审批、Artifact 上传、PR/MR 创建、敏感内容脱敏和权限拒绝。AuditLog 不应允许普通用户修改或删除。

⸻

12. 数据存储要求

TaskForge v0.1 使用关系型数据库作为业务事实来源，推荐 Postgres。核心表包括 projects、repositories、roadmap_items、requirements、findings、work_items、context_bundles、runner_profiles、agent_sessions、session_events、artifacts、comments、prompt_versions、audit_logs 和 outbox_events。SessionEvent 必须使用追加式表结构，支持按 session_id 和 seq 查询。Artifact 二进制内容可以存储在本地文件系统或对象存储中，数据库保存元数据。

平台应采用数据库事务保证启动 Session 的一致性。用户点击 Start 后，后端必须在同一事务中完成 WorkItem 锁定、权限校验、ContextBundle 创建或选择、AgentSession 创建、WorkItem active_session_id 更新、初始 SessionEvent 写入和 outbox dispatch 事件写入。事务提交后，异步 Dispatcher 再向 Local Runner 派发 ExecutionEnvelope。若派发失败，Session 状态必须从 Queued 或 Dispatching 转为 Failed 或 Awaiting Input，并产生对应事件。

第一版不使用 Kubernetes CRD 存储 Project、WorkItem、ContextBundle、RunnerProfile、AgentSession、SessionEvent、PromptVersion 或 Artifact。Kubernetes 可以作为未来部署或执行后端选项，但不参与 v0.1 业务对象建模。

⸻

13. Local Runner 技术要求

Local Runner 必须以命令行工具或常驻 Daemon 形式提供。推荐命令包括 login、register、bind-repo、start、status、doctor 和 logout。Runner 安装后，用户通过 login 连接平台，通过 register 创建 RunnerProfile，通过 bind-repo 将项目仓库绑定到本地路径，通过 start 启动常驻连接。

Runner 必须主动连接平台，建立 WebSocket 或 gRPC Stream。Runner 必须定期发送 heartbeat，并上报版本、Adapter、能力、本地绑定仓库和当前状态。Runner 接收 ExecutionEnvelope 后，必须执行本地预检，包括仓库路径存在性、Git remote 匹配、工作区清洁度、分支状态、Agent CLI 可用性、允许命令、禁止路径和本地权限策略。预检失败时不得启动 Agent，必须回传 runner.rejected 事件。

Codex CLI Adapter 是 v0.1 必须支持的 Adapter。Adapter 负责将 ExecutionEnvelope 转换为 Codex CLI 输入，调用 Codex CLI，采集 stdout/stderr、Agent 输出、命令结果和文件变更。Runner 不要求理解 Agent 内部推理，但必须记录可展示的执行摘要、命令、文件变更和结果。Runner 必须能在执行完成后生成 Patch Artifact，并可根据配置运行 verification commands。

⸻

14. 典型接口需求

平台 API 至少需要覆盖项目、仓库、需求、WorkItem、Finding、Runner、Session、Artifact 和事件流。具体接口形式可以由工程实现决定，但语义必须稳定。关键接口包括创建项目、获取项目空间、连接仓库、同步仓库元数据、创建需求、创建 WorkItem、更新 WorkItem 状态、编译 ContextBundle、注册 Runner、Runner heartbeat、启动 Session、停止 Session、获取 Session、追加 SessionEvent、获取 SessionEvent 流、上传 Artifact、创建 PR/MR 记录和写入 AuditLog。

Runner 与平台之间的协议必须围绕 ExecutionEnvelope 和 SessionEvent 设计。平台下发 ExecutionEnvelope，Runner 回传 SessionEvent 和 Artifact。Runner 不应直接修改 WorkItem 最终状态，而应通过 SessionEvent 和 Session 结果由平台服务进行状态转换。这样可以保证业务状态一致性和审计完整性。

15. MVP 范围

TaskForge v0.1 必须交付以下能力：项目空间、GitHub/GitLab 仓库连接、简单 Roadmap、简单需求清单、WorkItem 看板、WorkItem 详情页、ContextBundle v0、Local Runner 注册、Runner 心跳、本地仓库绑定、Codex CLI Adapter、/goal /plan /investigate 模式、ExecutionEnvelope 派发、SessionEvent 实时回传、Session Viewer、Patch Artifact、测试日志 Artifact、手动创建或记录 PR/MR、基础权限、敏感路径过滤、命令白名单和 AuditLog。

v0.1 可以延后但不应破坏架构的能力包括：复杂 Roadmap 管理、复杂需求拆解、双向 Issue 同步、Prompt Review、Skill Registry、Session Resume、Runner 任务队列、多 Runner 自动选择、云端执行、K8s Runner、企业 SSO、细粒度 RBAC、自动 PR Review、InferenceLab Pack 和跨项目知识检索。

16. 验收标准

TaskForge v0.1 的产品验收以端到端任务闭环为准。第一个验收场景是用户可以创建项目并连接 GitHub 或 GitLab 仓库，进入项目工作空间后看到 Roadmap、Requirements、Findings、Board、Sessions 和 Settings 标签页。第二个验收场景是用户可以在 Board 中创建或导入一个 Bug 类型 WorkItem，进入详情页后看到任务描述、验收标准、相关仓库、上下文包和执行控制区。第三个验收场景是用户可以在本机安装并启动 Local Runner，将项目仓库绑定到本地路径，并在平台看到 Runner 在线。第四个验收场景是用户在 WorkItem 页面选择 Local Codex CLI Runner，以 /goal 模式启动 Session，平台创建 AgentSession 并将 ExecutionEnvelope 下发至 Runner。第五个验收场景是 Runner 在本地调用 Codex CLI 执行任务，并实时回传 SessionEvent，其他团队成员可以在平台看到 Running Session、Agent 输出、命令日志、文件变更、Diff 和验证结果。第六个验收场景是 Session 完成后平台保存 Patch、日志、测试报告和总结，并将 WorkItem 转为 Needs Review 或展示失败原因。第七个验收场景是敏感路径和命令白名单生效，Runner 不允许上传 .env、secrets/** 或执行未授权命令。第八个验收场景是所有关键操作可在 AuditLog 中追溯。

⸻

17. 产品指标

v0.1 不以 Star 数或注册用户数作为核心验证指标。第一阶段重点观察端到端任务执行质量和团队协作价值。关键指标包括：从进入 WorkItem 到启动第一个 Session 的平均时间；ContextBundle 被用户认为足够支持执行的比例；Local Runner 成功接收并启动任务的比例；Session 正常完成率；Session 产生可 Review Patch 或 PR/MR 的比例；Session 失败后能够明确给出失败原因的比例；团队成员查看他人 Session 的次数；同一 WorkItem 多次 Session 的历史可追溯完整度；敏感内容误上传次数；用户手动补充上下文次数；WorkItem 从启动 Session 到进入 Needs Review 的转化率。

v0.1 成功标准为：在真实项目中，至少能够稳定完成“Bug WorkItem → ContextBundle → Local Codex Runner → Running Session → Patch Artifact → Needs Review”的完整闭环；至少 60% 的 WorkItem 在启动前能够生成可用 ContextBundle；至少 40% 的 /goal Session 能产出可 Review 的 Patch、PR/MR 或明确失败结论；Session 事件流、Artifact 和 AuditLog 能支撑团队成员复盘一次 Agent 执行过程。

⸻

18. 版本规划

v0.1 是 Local Runner MVP，目标是验证团队任务控制面和本地 Agent 执行会话可观测闭环。v0.2 可以增强 GitHub/GitLab Issue 双向同步、Finding 扫描、Prompt 版本管理、Session Replay、Runner 权限策略和 PR/MR 创建体验。v0.3 可以引入 Skill Registry、Prompt Review、历史 Session 复用、任务上下文自动补全和更丰富的 Adapter。v0.4 之后再评估云端 Runner、K8s Runner、Runner Pool、企业级 SSO/RBAC、Inference Decision Pack 和私有化部署增强。

版本推进必须遵循一个约束：任何新能力都必须服务 WorkItem、ContextBundle、Local Runner、AgentSession 和 Artifact 这条主链路。不能为了平台完整性提前引入大而全项目管理、云端执行基础设施或多 Agent 编排能力。

⸻

19. 风险与约束

第一类风险是 Local Runner 可用性。用户本地环境差异较大，可能出现 Agent CLI 未安装、仓库路径错误、依赖缺失、测试命令不可用、Git 工作区不干净、网络断开和权限不足等问题。产品必须通过 runner doctor、执行前预检、明确错误事件和本地日志降低排查成本。

第二类风险是上下文质量。若 ContextBundle 过弱，Agent 执行效果会退化为普通 Prompt；若上下文过多，又会增加噪音和 token 成本。v0.1 必须优先保证上下文结构清晰、来源可追溯、用户可编辑，而不是追求完全自动化。

第三类风险是安全边界。Local Runner 虽然避免了云端执行，但仍可能上传敏感日志、Diff 或错误信息。产品必须从第一版实现 deny path、secret masking、命令白名单和 Artifact 脱敏。默认策略必须保守。

第四类风险是产品容易退化为“看板加 Agent 按钮”。必须保证 WorkItem 详情页、ContextBundle、ExecutionEnvelope、SessionEvent 和 Artifact 是一等对象。没有上下文编译、实时会话和产物审计，产品就不成立。

第五类风险是范围膨胀。Roadmap、需求管理、扫描问题、知识沉淀和 Runner 扩展都有价值，但 v0.1 的唯一主路径必须是从 WorkItem 到 Local Runner Session 的执行闭环。任何无法增强该闭环的能力不得进入 P0。

⸻

20. 最终定版要求

TaskForge v0.1 的定版需求如下：产品是数据库驱动的 AI 原生工程协作工作台；第一版仅支持 Local Runner 执行，不做云端执行；平台负责项目、需求、看板、上下文、Runner 注册、Session 可观测、Artifact 和审计；Local Runner 负责本地仓库、Agent CLI、文件修改、命令执行、测试运行和 Patch 生成；WorkItem 是协作核心对象，ContextBundle 是执行上下文对象，AgentSession 是执行过程对象，SessionEvent 是可观测事件对象，Artifact 是结果沉淀对象。

第一版必须完成以下端到端能力：

Project Workspace
→ GitHub/GitLab Repository Link
→ Roadmap / Requirement / Finding / Board
→ WorkItem Detail
→ ContextBundle
→ Local Runner Selection
→ /goal Session Start
→ ExecutionEnvelope Dispatch
→ Local Codex CLI Execution
→ SessionEvent Streaming
→ Team-visible Session Viewer
→ Patch / Log / Test Artifact
→ PR/MR or Needs Review
→ Audit and Knowledge Retention

这就是 v0.1 的产品边界。不要做云端 Runner，不要做 K8s Operator 控制面，不要做通用项目管理平台，不要做新的 Coding Agent。TaskForge 第一版只做一件事：把团队工程任务变成本地 Agent 可以接手、团队可以观察、过程可以审计、结果可以复用的执行对象。
